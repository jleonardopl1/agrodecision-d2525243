## P0 — segurança / produto (precisa de decisão do dono)

### 1. `whatsapp-webhook` não valida HMAC (`X-Hub-Signature-256`)

`supabase/functions/whatsapp-webhook/index.ts` só valida o `hub.verify_token` no **GET**. No
**POST** (mensagens), o corpo não é verificado. A Meta assina cada POST com
`X-Hub-Signature-256 = HMAC-SHA256(app_secret, raw_body)`. Sem checar, um POST forjado com um
`from` que já existe em `chat_vinculos` (verificado) injeta produção/fixações/alertas na
carteira daquele cooperado. O Telegram **não** tem esse furo (valida o secret token no header).

**Proposta de fix** (precisa do secret `WHATSAPP_APP_SECRET` configurado antes de ativar):

```ts
// no topo:
const APP_SECRET = Deno.env.get("WHATSAPP_APP_SECRET") ?? "";

async function assinaturaValida(req: Request, raw: string): Promise<boolean> {
  if (!APP_SECRET) return false; // fail-closed: sem secret, rejeita
  const sig = req.headers.get("x-hub-signature-256") ?? "";
  const key = await crypto.subtle.importKey(
    "raw", new TextEncoder().encode(APP_SECRET),
    { name: "HMAC", hash: "SHA-256" }, false, ["sign"],
  );
  const mac = await crypto.subtle.sign("HMAC", key, new TextEncoder().encode(raw));
  const hex = [...new Uint8Array(mac)].map((b) => b.toString(16).padStart(2, "0")).join("");
  return sig === `sha256=${hex}`;
}

// no handler POST, ANTES de JSON.parse:
const raw = await req.text();
if (!(await assinaturaValida(req, raw))) return new Response("forbidden", { status: 403 });
const payload = JSON.parse(raw);
```

> ⚠️ Só ativar depois de configurar `WHATSAPP_APP_SECRET`, senão **todas** as mensagens são
> rejeitadas. Por isso é P0 com decisão do dono, não um fix automático.

### 2. Tensão: chatbot prescritivo × "nunca recomendação"

`CLAUDE.md` (inegociável): *"O bot entrega análise e simulação informativas — nunca
recomendação de compra/venda. Quem decide é o produtor."* Mas o `SYSTEM_BASE` do
`chatbot/index.ts` diz "entregue análises, alertas e **indicações de mercado**" e "na dúvida
entre vender e aguardar, **sugira venda PARCIAL**". Isso é recomendação.

**Proposta** (ajuste de tom, decisão do dono): trocar verbos prescritivos por informativos —
"mostre os números e o gatilho; deixe a decisão com o produtor"; remover "sugira venda
parcial"; substituir por "apresente o cenário de uma venda parcial **se o produtor perguntar**".
A coluna `sinais_ia.recomendacao` deve ser apresentada como leitura de mercado, não conselho.

## P1 — hardening (vira migration 0010 + ajuste de painel)

### 3. Funções `SECURITY DEFINER` expostas via RPC

Advisor `0028`/`0029`. Funções de **trigger** (`protect_cooperado_cols`,
`protect_cooperativa_cols`, `protect_relatorio_cols`) não precisam de `EXECUTE` para ninguém —
triggers rodam como dono da tabela. Helpers de RLS (`is_master`, `is_staff`, `is_coop_admin`,
`current_cooperativa_id`, `staff_has_permission`) são usados **dentro** das políticas, não
precisam estar no PostgREST. `get_coop_branding` é **intencionalmente** público (portal
co-branded) — manter.

**Proposta — `supabase/migrations/0010_hardening_rpc.sql`** (rascunho; aplicar via `db push`
só com aprovação):

```sql
-- Triggers: ninguém chama via API.
revoke execute on function public.protect_cooperado_cols()   from anon, authenticated, public;
revoke execute on function public.protect_cooperativa_cols() from anon, authenticated, public;
revoke execute on function public.protect_relatorio_cols()   from anon, authenticated, public;

-- Helpers de RLS: usados dentro das policies (SECURITY DEFINER), não pelo cliente.
revoke execute on function public.is_master()                       from anon, authenticated;
revoke execute on function public.is_staff()                        from anon, authenticated;
revoke execute on function public.is_coop_admin()                   from anon, authenticated;
revoke execute on function public.current_cooperativa_id()          from anon, authenticated;
revoke execute on function public.staff_has_permission(public.app_permission) from anon, authenticated;
-- get_coop_branding(text): mantém EXECUTE para anon (portal público) — intencional.
```

> Validar antes: as políticas RLS continuam funcionando porque chamam os helpers no contexto do
> servidor (o `revoke` tira só o acesso **direto via REST**, não o uso interno). Testar login +
> leitura de dados próprios em staging antes do remoto.

## P2 — performance de banco (não urgente)

- **`auth_rls_initplan` (~17 políticas):** trocar `auth.uid()`/`current_setting(...)` por
  `(select auth.uid())` nas políticas, para o planejador avaliar uma vez por query, não por
  linha. Reescrever no padrão ao mexer em cada migration.
- **`multiple_permissive_policies` (~46):** há múltiplas políticas permissivas para o mesmo
  par (role, ação) em várias tabelas (`staff_*`, `cooperados`, `commodities_config`, ...).
  Consolidar em uma política por ação quando possível.
- **`unindexed_foreign_keys`:** `revenue_share_events.cooperado_id`,
  `staff_group_members.group_id` (e afins) sem índice de cobertura — adicionar índice.
- **`pg_net` no schema `public`:** mover para schema próprio (advisor `0014`).
- **`unused_index` (~27):** **não** dropar agora — banco novo, sem tráfego real. Reavaliar
  depois de uso em produção.
